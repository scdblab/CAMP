Frequently Asked Questions

How is KOSAR different than memcached and other caching solutions?

KOSAR is different in that it reduces the complexity of the application software to empower software developers to focus on the requirements of the application. With memcached and other caching solutions, software engineers must manage idiosyncrasies of using the cache and provide software to address questions such as: How to maintain the cache consistent? What to do with undesirable race conditions? How to manage the software architecture with memcached client component? etc. This software must be designed, developed, debugged, tested, deployed, and maintained over time. KOSAR eliminates this overhead all together.

What is the advantage of using KOSAR when compared with memcached?

KOSAR is faster to deploy and reduces the complexity of application software. It enhances the productivity of software developers by empowering them to focus on the requirements of their application. KOSAR provides a higher performance than memcached because it employs an intelligent memory management technique (CAMP).

Do I have to change my application software to use KOSAR?

No; KOSAR is a transparent caching solution. This means it requires no change to the application software. One simply replaces an existing RDBMS client component (such as a JDBC jar file) with KOSAR and the application is ready to be launched.

Do I have to re-design my database to use KOSAR?

NO; same response as the question on whether the application software must change.

What does KOSAR cache?

KOSAR caches a SQL query instance and its result set. An application may issue millions and billions of cached query instances and KOSAR manages these intelligently while providing strong consistency.

Does KOSAR use the concept of Time To Live (TTL) for the cached entries?

No. KOSAR detects when the result of a cached query instance has changed in a transaction manner, avoiding the use of techniques such as TTL.

What RDBMSs are supported?

KOSAR functions with almost all SQL systems such as Oracle, MySQL, Microsoft SQL Server, IBM DB2, PostgreSQL, and others.

Does KOSAR work with an ORM such as Hibernate?

Yes. Its deployment is as trivial as replacing the existing driver name in the cfg file with KOSAR's driver name and restarting the application.

Does KOSAR support the transaction guarantees of the RDBMS?

Yes, as long as the underlying RDBMS is configured to provide transactional guarantees, KOSAR preserves these guarantees. For example, with MySQL 5.6, tables must be identified as transactional at creation time. Once this is done, KOSAR preserves the transactional properties of MySQL.

How does one check to see if KOSAR is operational?

KOSAR has a built-in web server that supports a monitoring tool. Visit the Demo tab and click on the Live Demo hyperlink. Using the monitoring tool, one may turn KOSAR on and off in real time. Moreover, one may identify the application's query templates cached by KOSAR.

What is the difference between a query template and a query instance?

The where clause of a query identifies it either as a query instance or a query template. A query template uses wildcards while a query instance uses values. For example, the where clause of a query instance might be Member.userid=5. The query template for this instance would be Member.userid=?. An application may issue millions and billions of unique query instances for a given query template.

Why is the distinction between a query template and a query instance important?

KOSAR enables an administrator to use its monitoring tool to disable caching either at the granularity of a query template or a query instance.

If I disable the caching of a query template, does it stop caching instances of the query template?

Yes.

If I disable the caching of a query template, does KOSAR delete its cached instances?

No. These instances are maintained up to date and used once the caching of the query template is re-enabled. To delete these instances, press the flush button for the query template.
